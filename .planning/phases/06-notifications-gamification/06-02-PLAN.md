    - `export async function processOverdueNotifications(now: Date = new Date()): Promise<number>`:
      - `const pb = await createAdminClient();` (D-04, T-06-02-01 defense — admin client needed to iterate all homes; bypasses owner-only rules).
      - Fetch all homes: `pb.collection('homes').getFullList({ fields: 'id,timezone' })`.
      - For each home:
        - Fetch members: `pb.collection('home_members').getFullList({ filter: \`home_id = "${home.id}"\`, expand: 'user_id', fields: 'id,role,user_id,expand.user_id.id,expand.user_id.ntfy_topic,expand.user_id.notify_overdue,expand.user_id.name' })`.
        - Filter members to those with `ntfy_topic !== ''` AND `notify_overdue === true` (per-user opt-in + topic configured).
        - If no eligible members, skip home.
        - Fetch tasks: `pb.collection('tasks').getFullList({ filter: \`home_id = "${home.id}" && archived = false\`, fields: 'id,home_id,name,frequency_days,schedule_mode,anchor_date,created,archived' })`.
        - Fetch completions via `getCompletionsForHome(pb, taskIds, now)` from lib/completions.ts.
        - Reduce: `const latestByTask = reduceLatestByTask(completions)`.
        - For each task:
          - Compute nextDue via `computeNextDue(task, latestByTask.get(task.id) ?? null, now)`. If null (archived — shouldn't happen given filter) or `nextDue > now`, skip.
          - Task is overdue. Build `refCycle = buildOverdueRefCycle(task.id, nextDue.toISOString())`.
          - For each eligible member: `if (await hasNotified(pb, member.user.id, refCycle)) continue;` then `const ntfyResult = await sendNtfy(process.env.NTFY_URL || 'https://ntfy.sh', member.user.ntfy_topic, { title: \`Overdue: ${task.name}\`, body: \`Your ${home.name ? home.name + ' ' : ''}${task.name.toLowerCase()} is overdue — ready when you are.\`, priority: 3, tags: ['hourglass'] });` — note we don't block-on-success: write the notifications record regardless to dedupe the NEXT tick. Even on network fail we don't retry; better to miss one ping than spam.
          - `await recordNotification(pb, { user_id: member.user.id, home_id: home.id, task_id: task.id, kind: 'overdue', sent_at: now.toISOString(), ref_cycle: refCycle });`
          - Increment counter on ntfyResult.ok && record written.
      - Return total count sent.

      - Fetch all homes. For each home:

          - weekStartIso = fromZonedTime(startOfWeek(zonedNow), home.timezone).toISOString().
          - refCycle = buildWeeklyRefCycle(member.user.id, weekStartIso).

          - `const summary = computeWeeklySummary(completions, tasks, areas, now, home.timezone);`
          - Compose body: `"This week: ${summary.completionsCount} tasks. ${home.name || 'The house'} is ${summary.coveragePercent}% maintained.           - `await recordNotification(pb, { user_id: member.user.id, home_id: home.id, task_id: null, kind: 'weekly_summary', sent_at: now.toISOString(), ref_cycle: refCycle });`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron tick → PB | scheduler uses admin client (D-04, T-06-02-01) to iterate homes; inserts via createRule=null — only admin writes allowed |
| updateTask server action → scheduler | user-authed action triggers admin-client ntfy fire; the assignee user_id is sourced from VALIDATED task formData (passes PB relation rule before ntfy fires) |
| completeTask server action → scheduler | same; completer id is server-sourced from authStore |
| Admin trigger route → internet | POST /api/admin/run-scheduler gated by ADMIN_SCHEDULER_TOKEN (32+ chars) |
| HomeKeep → ntfy.sh | outbound HTTPS POST only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-02-01 | Elevation of privilege | scheduler admin-client use | mitigate | createAdminClient requires PB_ADMIN_EMAIL/PASSWORD env — same secret Phase 4 invite-accept depends on. Not exposed to client. Scheduler module is server-only (dynamic-imported from instrumentation + server actions). |
| T-06-02-02 | Tampering | updateTask assigned-notification spoofing | mitigate | previousAssignedToId sourced via `pb.collection('tasks').getOne(taskId)` which enforces the tasks.viewRule; assignee id is validated by PB's relation-field rule before the update commits. Client cannot cause a notification to fire for a task they don't own. |
| T-06-02-03 | Information disclosure | partner-completed body leaks task name | accept | Task names within a shared home are already visible to co-members per 04-03 users.viewRule. Notification just mirrors in-app visibility. |
| T-06-02-04 | Denial of service | scheduler unbounded home iteration | accept | Single-container v1 deploy; household scale. A 1000-home deploy would need leader election (deferred — see CONTEXT §D-08). Admin route is token-gated so arbitrary triggers cannot amplify. |
| T-06-02-05 | Elevation of privilege | /api/admin/run-scheduler without token | mitigate | Route returns 401 when ADMIN_SCHEDULER_TOKEN unset OR length <32 OR header mismatch. Fail-closed. |
