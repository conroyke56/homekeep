    **lib/weekly-summary.ts** (D-12, GAME-03):
    - `export type WeeklySummary = { completionsCount: number; coveragePercent: number; topArea: string; mostNeglectedTask: { id: string; name: string; daysOverdue: number } | null };`
    - `export function computeWeeklySummary(completions: CompletionRecord[], tasks: Task[], areas: Array<{ id: string; name: string }>, now: Date, timezone: string): WeeklySummary`
    - Steps:
      1. Compute weekStart = fromZonedTime(startOfWeek(toZonedTime(now, timezone)), timezone).
      2. `completionsCount = completions.filter(c => new Date(c.completed_at) >= weekStart).length`.
      3. `latestByTask = reduceLatestByTask(completions)`.
      4. `coveragePercent = Math.round(computeCoverage(tasks, latestByTask, now) * 100)`.
      5. topArea: per-area, count completions in this week whose task.area_id === area.id; pick area with MAX count. Tie-break alphabetically by name. If no completions this week, topArea = areas[0]?.name ?? 'No area'.
      6. mostNeglectedTask: for each active task, compute daysOverdue = max(0, (now - computeNextDue(task, latest, now)) / 86400000). Pick MAX. Tie-break by task.created DESC (newer wins). If no overdue tasks or no tasks, null.
    - Imports: `computeCoverage` from `@/lib/coverage`, `computeNextDue` from `@/lib/task-scheduling`, `reduceLatestByTask` from `@/lib/completions`.
    - Test matrix: (1) empty inputs → `{completionsCount:0, coveragePercent:100, topArea:'No area', mostNeglectedTask:null}` (empty-home invariant holds); (2) 3 completions this week spread across 2 areas (Kitchen:2, Bathroom:1) → topArea='Kitchen'; (3) 1 overdue task (7d overdue on 1-day freq → clamped health=0), 1 on-time task → coveragePercent=50; (4) most-neglected = task with 14d overdue wins over 3d overdue; (5) tie on daysOverdue → newer created wins; (6) tasks empty → mostNeglectedTask null and coverage=100; (7) completionsCount excludes completions from LAST week (completion at weekStart-1d); (8) DST-safe week boundary (Melbourne AEDT→AEST).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser ↔ PB | users read their OWN notifications rows only (listRule/viewRule = user_id = auth) |
| scheduler (server) → PB | scheduler uses admin client to insert notifications; regular users cannot write (createRule=null) |
| HomeKeep ↔ ntfy.sh | outbound-only HTTPS POST; no reply body processed beyond HTTP status |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | Spoofing | notifications.viewRule | mitigate | `user_id = @request.auth.id` — user cannot read another's notification history. Proven by integration test: Alice rows not visible to Bob via PB filter. |
| T-06-01-02 | Tampering | notifications writes | mitigate | createRule=null + updateRule=null + deleteRule=null — only superuser (scheduler via createAdminClient) can mutate. Integration test proves regular-user PATCH + DELETE reject. |
| T-06-01-03 | Repudiation | notification audit trail | mitigate | append-only contract via null write rules; duplicate inserts blocked by (user_id, ref_cycle) unique index. |
| T-06-01-04 | Information disclosure | ntfy log output | mitigate | `lib/ntfy.ts` never logs `msg.title`, `msg.body`, or `topic` — only the HTTP status/error code on failure. Topic is per-user secret (D-01). |
| T-06-01-05 | Information disclosure | PB ntfy_topic field | accept | users table viewRule already `id = @request.auth.id || home_members_via_user_id.home_id.home_members_via_home_id.user_id ?= @request.auth.id` (04-03 migration) — co-members CAN see each other's ntfy_topic. Accepted because (a) topic is a per-user push-target, not a secret shared across households, and (b) the threat model for phase 4 household-sharing already accepts social-level visibility. Phase 7 hardening could add a field-level filter to strip ntfy_topic from expand responses if this becomes a concern. |
| T-06-01-06 | Denial of service | ntfy rate limit | accept | ntfy.sh rate-limits are generous (60/min/topic public) and the scheduler runs hourly — a household with 100 overdue tasks still only fires 100 POSTs/hour per user. No client-side limit added in 06-01. |
| T-06-01-07 | Denial of service | unique-index race | mitigate | Wave 2's scheduler does pre-check via `hasNotified()` to avoid sending ntfy for already-notified rows; the unique index is the race-condition safety net. Duplicate insert returns null, no exception propagates. |
| T-06-01-08 | Information disclosure | weekly summary payload | accept | summary body contains task count + coverage % + top area name + most-neglected task name.
