## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Next.js/SDK → PB HTTP API | All data mutations cross here; PB API rules are the trust boundary |
| Container env → PB runtime (`onBootstrap`) | SMTP credentials + rate-limit config read from process env at PB start |
| PB DB transaction → `onRecordCreateExecute` hook | Hook runs inside same transaction as the triggering insert; throws roll back |

|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Elevation | homes/areas/tasks collections with forged `owner_id` | mitigate | PB API rules `createRule: '@request.auth.id != "" && owner_id = @request.auth.id'` block client-supplied owner_id mismatch (RESEARCH §Security Domain). Areas + tasks use `home_id.owner_id = @request.auth.id` subquery. Verified in migration per D-09/D-10/D-12. |
| T-02-01-02 | Tampering | AREA-02 bypass — user tries to delete Whole Home | mitigate | Schema-level `deleteRule: '… && is_whole_home_system = false'` (open question #4 resolution: "both"). UI-level guard comes in 02-04. |
| T-02-01-03 | Denial of Service | Login brute force | mitigate | `bootstrap_ratelimits.pb.js` enables PB rate limits with `*:auth-with-password` rule at 5/60s for `@guest` audience (RESEARCH §Security Domain line 1778). |
| T-02-01-04 | Information Disclosure | SMTP password in PB logs | accept | PB logs the host/port/user but never the password (verified in `bootstrap_smtp.pb.js` — `console.log` line excludes `pass`). PII-free. |
| T-02-01-05 | Tampering | Hook fails → orphan home row | mitigate | `onRecordCreateExecute` runs inside the same DB transaction as the home insert (per RESEARCH design note + Pitfall 7); any throw rolls the whole thing back.
