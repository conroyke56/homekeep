<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form POST → Server Action | Form data crosses trust boundary; server action must zod-validate |
| Server Action → PocketBase `users` collection | Auth credentials and user creation traverse PB API with built-in bcrypt |
| proxy.ts → pb_auth cookie | Cookie value is ONLY checked for presence, not verified cryptographically |
| Browser cookie jar ↔ pb_auth | HttpOnly + SameSite=Lax + Secure(prod) — JS cannot read; cross-site forms cannot submit |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Spoofing | User forges pb_auth cookie to impersonate another user | mitigate | proxy.ts only checks cookie presence; PB server validates the JWT on every API call. Forged cookie → first pb.collection().getList() call fails → Server Component redirect catches. Defense in depth via `(app)/layout.tsx` Server Component guard checking `pb.authStore.isValid`. |
| T-02-03-02 | Tampering / CSRF | Cross-site form posts loginAction | mitigate | Next 16 server actions embed an encrypted action ID (RESEARCH §Security Domain); SameSite=Lax on pb_auth blocks cross-site cookie transmission on form posts. |
| T-02-03-03 | Info Disclosure | Password error reveals whether email is registered | mitigate | loginAction returns generic "Invalid email or password"; requestResetAction always returns `{ok: true}` on success regardless (RESEARCH §Server Action). |
| T-02-03-04 | Info Disclosure | XSS reads pb_auth via document.cookie | mitigate | Cookie set with `httpOnly: true` in cookieOptions() (RESEARCH Pitfall 3); secure flag in production. |
| T-02-03-05 | Denial of Service | Login brute force | mitigate | Rate limited by 02-01 `bootstrap_ratelimits.pb.js` at 5/60s for @guest. This plan's proxy.ts does no additional throttling — delegate to PB. |
| T-02-03-06 | Info Disclosure | Stack trace or PB internal error leaked in formError | mitigate | Actions catch all PB errors and return sanitized strings (RESEARCH §Security Domain V7 + server-action code). Never return `err.message` raw. |
| T-02-03-07 | Tampering | Unencrypted cookie transmission in HTTP LAN dev | accept | `secure: production` means LAN-HTTP deployments don't set Secure; SameSite=Lax mitigates most CSRF. Phase 7 adds HTTPS modes (RESEARCH Pitfall 3 "Open concern for Phase 7"). |
| T-02-03-08 | Elevation | Server action reads `next` query param → open redirect | mitigate | Current code in loginAction uses `redirect('/h')` — ignores `next`. | T-02-03-09 | Info Disclosure | Signup action leaks email via fieldErrors.email "already registered" | accept | This is explicit UX — the friction of guessing emails via signup is a known tradeoff and the alternative (generic error) degrades signup UX noticeably.
