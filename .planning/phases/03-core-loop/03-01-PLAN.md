<pitfalls_acknowledged>
The following research pitfalls MUST be addressed in this plan's code + tests:

- Pitfall 2 (Timezone day-boundary off-by-one): computeTaskBands uses `fromZonedTime(startOfDay(toZonedTime(now, tz)), tz)` — unit test MUST cover a Melbourne+UTC straddle case.
- Pitfall 3 (PB getFullList default batch size): getCompletionsForHome passes explicit `batch: 500`.
- Pitfall 5 (Server-action throw vs error-shape return): completeTaskAction returns typed results for business cases (requiresConfirm, archived, not-signed-in). Only network/PB outages propagate as `{ok:false, formError}`.
- Pitfall 6 (revalidatePath + router.refresh complementary): this plan calls `revalidatePath` in the action; 03-03 adds `router.refresh()` on the client.
- Pitfall 11 (PB double-hop filter slow without index): migration creates `CREATE INDEX idx_completions_task_completed ON completions (task_id, completed_at)` AND `CREATE INDEX idx_completions_completed_at ON completions (completed_at)`.
- Pitfall 13 ("Via" field required): server action hardcodes `via: 'tap'`; schema defaults to 'tap'.
- Anti-pattern "PB updateRule = '' for append-only": use `null`, not empty string.
- Anti-pattern "Reading Date.now() inside pure function": all four pure functions accept `now: Date` as a parameter; never read the clock internally.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server action | Untrusted taskId + force flag crosses here |
| server action → PocketBase API | Authenticated, but PB rules enforce ownership |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Spoofing | completeTaskAction | mitigate | Ownership preflight via `pb.collection('tasks').getOne(taskId)` — PB viewRule on tasks requires `home_id.owner_id = @request.auth.id`; cross-user forged ids 404 before the write. |
| T-03-01-02 | Tampering | completions.create payload | mitigate | Server action hardcodes `completed_by_id: userId` from authStore, NEVER from client. PB createRule also enforces `@request.body.completed_by_id = @request.auth.id` as defense-in-depth. |
| T-03-01-03 | Repudiation | append-only completions | mitigate | updateRule=null + deleteRule=null locks the collection (per PB docs, null = superusers-only). Integration test proves PATCH + DELETE return 403/400 for authenticated non-superuser callers. |
| T-03-01-04 | Information Disclosure | list/view rules | mitigate | listRule + viewRule: `@request.auth.id != "" && task_id.home_id.owner_id = @request.auth.id` — double-hop enforces home-owner scope. |
| T-03-01-05 | Denial of Service | rapid completeTaskAction spam | accept | Phase 2 PB `/api/` rate limit (300/60s) remains; no per-user throttle added in Phase 3. Revisit in Phase 7 ops hardening per RESEARCH Open Question #4. |

| T-03-01-08 | Spoofing | completing an archived task | mitigate | Server action reads `task.archived` post-preflight and returns `{ok:false, formError:'Task is archived'}` before attempting the write. |
