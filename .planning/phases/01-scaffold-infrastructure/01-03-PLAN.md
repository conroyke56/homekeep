<objective>
Author the s6-overlay v3 service tree under `docker/s6-rc.d/` that supervises three processes (Caddy, PocketBase, Next.js) inside the production container. Delivers the INFR-01 process-supervision requirement per D-01, implementing RESEARCH.md §Pattern 2 exactly.

Purpose: Without this tree, the Dockerfile from 01-02 cannot boot -- s6-overlay reads `/etc/s6-overlay/s6-rc.d/` on startup and refuses to enter stage 2 if the `user` bundle has no `contents.d/` entries. The three services run in parallel under s6 supervision, which gives us auto-restart on crash, SIGTERM propagation for clean SQLite shutdown (Pitfall #1 + #4 mitigation), and zombie reaping. Services run as `node` user via `s6-setuidgid` so PocketBase and Next.js can write to `/app/data` without being root (Pitfall #9).

Output: Twelve files under `docker/s6-rc.d/` implementing three longrun services and their registration in the default `user` bundle.

s6 service directory shape (for each of caddy, pocketbase, nextjs):
```
<service>/
  type            # text file with single word: "longrun"
  run             # executable script starting with `#!/command/with-contenv sh`
  dependencies.d/
    base          # empty file -- depends on s6 base bundle
```

User bundle registration (enables service in default bundle):
```
user/
  contents.d/
    caddy        # empty file
    pocketbase   # empty file
    nextjs       # empty file
```

Run-script contract per RESEARCH.md §Pattern 2:
- Shebang: `#!/command/with-contenv sh` (s6-overlay helper that keeps container env vars visible)
- Body: `exec <binary> <args>` (exec replaces the shell -- s6 supervises the binary, signals propagate, PID layout is correct)
- NOT allowed: backgrounded `&`, subshells, trap handlers, `while` loops (s6 does all of that for us)

Runtime flags required:
- Caddy: `run --config /etc/caddy/Caddyfile --adapter caddyfile`
- PocketBase: `serve --http=127.0.0.1:8090 --dir=/app/data/pb_data --migrationsDir=/app/pb_migrations`
- Next.js: `node server.js` from CWD `/app` (Dockerfile sets WORKDIR)

User privilege pattern (Pitfall #9):
- Caddy: root (binds 0.0.0.0:3000 inside container; no port-privilege issue on unprivileged binds >1024)
- PocketBase: `s6-setuidgid node` (writes to /app/data owned by node:node)
- Next.js: `s6-setuidgid node` (writes to /app/.next/cache as node)
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| s6 init (PID 1, root) -> supervised services | Signal and privilege dropping happens here |
| Container filesystem -> running services | UID 1000 (node) must own /app/data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-01 | Elevation of Privilege | PocketBase running as root | mitigate | `exec s6-setuidgid node` drops privileges before pocketbase exec'd |
| T-01-03-02 | Elevation of Privilege | Next.js running as root | mitigate | `exec s6-setuidgid node` drops privileges before node exec'd |
| T-01-03-03 | Elevation of Privilege | Caddy running as root | accept | Caddy is a static Go binary with minimal attack surface; bind to :3000 unprivileged so root not strictly required, but leaving root avoids s6-setuidgid complexity for the ingress tier; revisit in Phase 7 hardening |
| T-01-03-04 | Denial of Service | A crashed service is not restarted | mitigate | s6-overlay v3 `longrun` services are auto-restarted by default on exit (non-zero exit triggers restart); no extra configuration needed |
| T-01-03-05 | Information Disclosure | SIGTERM during a PocketBase SQLite write leaves WAL in bad state | mitigate | s6 propagates SIGTERM to all services, PocketBase handles SIGTERM with WAL checkpoint (upstream behavior); verified in RESEARCH.md §Pattern 2 "Signal handling" |
| T-01-03-06 | Tampering | A malicious migration file gets auto-applied | accept (phase 1) | `/app/pb_migrations` is empty in phase 1 and copied from image layer (immutable); phase 2 populates via source control and PR review |
</threat_model>
