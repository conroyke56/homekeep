<interfaces>
<!-- Caddy routing contract -->
<!-- Upstreams are loopback by design (Caddy, Next.js, PB all in the same container) -->

Listen address: `:3000` (single exposed port; Docker EXPOSE 3000 in 01-02 Dockerfile)

Routing rules (source order matters -- Caddy matches top-to-bottom):
1. `handle /api/health` -> `reverse_proxy localhost:3001` (Next.js, no flush override needed -- short GET)
2. `handle /api/*`      -> `reverse_proxy localhost:8090 { flush_interval -1 }` (PocketBase, SSE-safe)
3. `handle /_/*`        -> `reverse_proxy localhost:8090 { flush_interval -1 }` (PocketBase admin UI)
4. `handle`             -> `reverse_proxy localhost:3001` (Next.js catch-all for pages and /_next/static/)

Why `flush_interval -1`: PocketBase realtime uses Server-Sent Events. Caddy's default buffering batches the response; `-1` disables the write buffer so each `data: ...\n\n` frame is flushed immediately (RESEARCH.md §Critical Design Choice and Pitfall #10).

Why Next.js catches `/_next/static/`: Next.js standalone `server.js` serves its own static assets (RESEARCH.md §Architectural Responsibility Map); the catch-all lets it do that.

Why no `auto_https off`: the global Caddyfile directive is needed to prevent Caddy from trying to provision TLS on `:3000`. Phase 1 is HTTP-only on the internal port; Phase 7 introduces external TLS via the Caddy compose variant.
</interfaces>

```
{
	auto_https off
	admin off
}

:3000 {
	handle /api/health {
		reverse_proxy localhost:3001
	}

	handle /api/* {
		reverse_proxy localhost:8090 {
			flush_interval -1
		}
	}

	handle /_/* {
		reverse_proxy localhost:8090 {
			flush_interval -1
		}
	}

	handle {
		reverse_proxy localhost:3001
	}
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External client -> Caddy :3000 | All ingress traffic lands here |
| Caddy -> loopback upstreams | Internal reverse proxy; upstreams on 127.0.0.1 only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Tampering / SSRF | User-controlled path routed to attacker upstream | mitigate | Upstreams are hardcoded `localhost:3001` and `localhost:8090`; no path templating, no user input in upstream resolution (RESEARCH.md §Security Domain: "Caddy SSRF via crafted paths = Safe") |
| T-01-04-02 | Information Disclosure | Caddy admin API exposed | mitigate | `admin off` in global options block disables Caddy's :2019 admin endpoint entirely |
| T-01-04-03 | Information Disclosure | /_/* PB admin reachable from LAN | accept | Per D-05, PB admin at /_/ is intentionally accessible in production; PB has its own rate-limited admin auth; future OAuth configurable via PB admin UI |
| T-01-04-04 | Spoofing | Health endpoint could be gamed by upstream lying | mitigate | `/api/health` is routed to Next.js which does its own fetch to PB on loopback; PB cannot lie to Next.js about its own status -- a dead PB leaves Next's fetch unreachable and health returns 503 (Pitfall #4 + #8 mitigated) |
| T-01-04-05 | Denial of Service | SSE connections buffered, appearing broken | mitigate | `flush_interval -1` on both `/api/*` and `/_/*` routes disables Caddy's output buffering for PB streams; Pitfall #10 mitigation |
| T-01-04-06 | Elevation of Privilege | Accidental TLS provisioning on internal port | mitigate | `auto_https off` in global options disables Caddy's automatic HTTPS (would otherwise try to get a cert and spam logs) |
</threat_model>
