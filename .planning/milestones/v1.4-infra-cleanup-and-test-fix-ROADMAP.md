## Phase execution order rationale

**39 first (Claude-side, no gates).** TESTFIX-07 is purely
application code — startable today (2026-04-29). Gets us a head
start while the infra timers cook. Also serves as the v1.4 "soak
generator" — every commit ticks the 10-consecutive-CI counter
toward the v1.3 success criterion that's still naturally baking in.

**40 next (root-coordinated, timer-fired).** Wildcard TLS rebuild
goes after the 2026-05-01 timer drops
`/opt/vps/FOLLOWUP-2026-05-01.md`. Claude drafts the change set +
root prompt; user executes; Claude verifies. Required BEFORE 41 in
case the verify-revproxy-stable check needs to validate the new
build (defense in depth — actually 40 and 41 could parallelize on
the day, but doing them serially keeps the rollback story clean).

**41 last (cleanup + timer disable).** The `/home/sprout/revproxy/`
removal needs to happen AFTER 40 confirms the new revproxy is
healthy. Same root window — both timer-fired prompts can be
processed in one root session. Removes the timer units once both
FOLLOWUP files are processed.

## Coordination model

- **Phase 39 = pure Claude work.** Standard `/gsd-plan-phase 39` →
  `/gsd-execute-phase 39` flow.
- **Phases 40 + 41 = Claude drafts, root executes.** Each phase
  produces a `docs/vps-setup/<phase>.md` change set + a paste-ready
  root prompt. User runs as root, pastes the report URL back, Claude
  verifies + closes the phase.
