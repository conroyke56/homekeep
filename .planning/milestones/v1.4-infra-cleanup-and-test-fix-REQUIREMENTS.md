### TESTFIX-07 — Root-cause + fix `notifications.spec.ts` checkbox hydration

**Currently:** Both Part 1 (line 177) and Part 2 (line 251) of
`tests/e2e/notifications.spec.ts` are `test.skip()`'d with detailed
TODO comments deferring to "v1.4-notifications-checkbox-hydration."

**Background:** v1.3 made three test-plumbing fix attempts
(`.check()`, click+poll, click+generous-wait). All three failed. CI
log on the third attempt showed Playwright polling the checkbox
state 14 times over 10s with the input stuck at unchecked AFTER the
click event fired. Source: `f04df45` → `ffedf14` → `a3b7c7c` →
`5bf1108` (re-skip).

**Real root cause hypotheses (need verification):**

- **H1 — React 19 concurrent hydration race.** The notifications
  page renders an RHF `<form>` whose `register('weeklySummary')`
  call attaches the `onChange` handler during hydration. Headless
  Chromium's click event arrives BEFORE the handler is attached, so
  the click is swallowed.
- **H2 — RHF `register` vs Controller mismatch.** Plain
  `<input type=checkbox {...register('weeklySummary')} />` uses
  uncontrolled state under RHF, which interacts badly with React 19
  concurrent rendering on hydration boundary.
- **H3 — `useFormContext` provider re-mount.** The form provider
  remounts on data fetch settle; click happens during the remount
  window.

**Fix approach:** component-level investigation, NOT more test
plumbing.

1. Add a `data-hydrated="true"` signal to the form root that flips
   only AFTER `useEffect(() => setHydrated(true), [])` fires.
2. If H1 confirms (most likely), change the checkbox to
   `Controller`-wrapped `<Checkbox>` (shadcn) so RHF owns the value
   transition rather than the DOM.
3. If still flaky after (1)+(2), bisect with React 18 → confirms vs
   refutes "React 19 concurrent" as the proximate cause; if React 18
   stable, file upstream and ship the Controller fix as workaround.
4. Re-enable both `test.skip()` lines + run 10 CI cycles to confirm
   stable.

**Done when:** both `notifications.spec.ts` blocks un-skipped, run
green on CI 10 runs consecutively with **zero retries consumed** by
the v1.3 flake-budget gate.

**Estimated effort:** 1.5–3 hours code + investigation, plus 25 min
× 3 CI cycles soak = roughly half a day.

**Status (Phase 39 shipped 2026-04-29 — partial):** hydration race
fixed at component level (5-line `useState`+`useEffect` flip pattern
in `components/notification-prefs-form.tsx`). Part 1 un-skipped and

### INFRA-01 — Wildcard TLS via DNS-01 (caddy-dns/godaddy plugin rebuild)

**Currently:** `revproxy-caddy-1` issues per-subdomain certs via
HTTP-01 (port 80). Works but slower to provision new subdomains and
won't cover wildcards. Tracked by
`vps-followup-20260501.timer` (fires 2026-05-01 10:00 UTC; will drop
`/opt/vps/FOLLOWUP-2026-05-01.md`).

**Background:** Phase 34 decoupled revproxy to `/opt/vps/revproxy/`
with the upstream `caddy:2-alpine` image. To get `*.the-kizz.com`
wildcard via DNS-01 we need a custom Caddy build with the
`caddy-dns/godaddy` community plugin compiled in.

**Fix:**

1. Replace `image: caddy:2-alpine` in `/opt/vps/revproxy/docker-
   compose.yml` with a `build: ./caddy-godaddy/` directive pointing
   at a 5-line Dockerfile:

   ```Dockerfile
   FROM caddy:2-builder AS builder
   RUN xcaddy build --with github.com/caddy-dns/godaddy
   FROM caddy:2-alpine
   COPY --from=builder /usr/bin/caddy /usr/bin/caddy
   ```

2. Add `GODADDY_API_KEY` + `GODADDY_API_SECRET` env vars to
   `/opt/vps/revproxy/.env` (sourced from `/etc/secrets/godaddy.creds`
   via secrets group).
3. Update Caddyfile global block:

   ```
   {
       acme_dns godaddy {env.GODADDY_API_KEY} {env.GODADDY_API_SECRET}
   }
   ```

4. `docker compose -p revproxy build && docker compose -p revproxy
   up -d` — should issue `*.the-kizz.com` on first start.
5. Verify cert contains `*.the-kizz.com` SAN; `homekeep.demo.the-
   kizz.com` continues to resolve TLS without a per-subdomain cert.
6. Document the new build flow in `/opt/vps/revproxy/README.md`.

**Coordination:** This phase requires root permissions on
`/opt/vps/revproxy/`. The `homekeep` user (Claude) drafts the
Dockerfile + Caddyfile change set + a paste-ready root prompt in
`docs/vps-setup/wildcard-tls-rebuild.md`. User runs the prompt as
root.

**Done when:**

- `*.the-kizz.com` cert visible in `caddy data` directory
- `homekeep.demo.the-kizz.com` continues to serve via wildcard
  (curl validates SAN list)
- New subdomain (e.g. test record) provisions instantly via DNS-01
  without port-80 hit
- README documents rebuild flow

**Estimated effort:** Claude drafts 30 min; root execution + verify
30 min.
